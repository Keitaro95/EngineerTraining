TanStack QueryのSSRハイドレーションの流れを図で説明しますね。図のポイントを補足すると、**左のサーバー側の棚（青）と右のブラウザ側の棚（紫）は別物**です。同じ `QueryClient` という型でも、メモリ空間が完全に分かれているので、直接渡すことはできません。

そこで真ん中の **HTML（オレンジ）が橋渡し役**になります。サーバー側で棚の中身を `dehydrate()` で JSON に変換して HTML に埋め込み、ブラウザ側で `HydrationBoundary` がその JSON を見つけて、空っぽの棚に `hydrate` する、という流れです。

結果として `PokemonInfo` が `useSuspenseQuery` を呼んだときには、**棚にはもうデータが入っている状態**になるので、API を再度叩くことなく即座にレンダリングできる、というのがこの仕組みのメリットです。
